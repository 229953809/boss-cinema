# EXO 高码率动态网络保护、自适应降速与保调播放最佳实践方案

日期：2026-07-22

实施基线：`main-2 / playback-optimization-item-62`

实施结果：`playback-optimization-item-63`（代码、单元测试和 ARM64 双变体构建已完成）

Media3：`1.11.0-alpha01-fongmi`

适用场景：4K/5K、HEVC/H.265、大体积 MKV/REMUX、VBR 高峰码率、网盘 Range、Go 二进制代理、`127.0.0.1` 本地播放链路、持续吞吐略低于媒体消费速度。

## 一、审阅后的最终结论

该能力不能实现为“发生异常后固定减速”，也不能实现为“按固定小步长慢慢降到下限”。正确模型是：

```text
休眠，不改变任何已有功能
  → 观察缓冲水位、快慢趋势和加载状态
  → 估算当前可支撑播放倍率
  → 预测到达安全缓冲线的剩余时间
  → 判断降速是否真的能够在 0.85～1.00x 内稳定播放
  → 计算连续目标倍率和必须在何时到达该目标
  → 使用期限驱动的动态斜坡靠近目标
  → 网络恢复后平滑回到 1.00x
  → 自动退出并重新观察
```

速度区间分成两层：

| 区间 | 定义 | 使用条件 |
|---|---|---|
| `0.97～1.00x` | 优先轻量保护区 | 持续轻微缺口，正常确认后介入 |
| `0.85～0.97x` | 深度保护扩展区 | 只有算法证明轻量区不足、目标倍率仍不低于 0.85x、且能在缓冲耗尽前到达目标时才允许进入 |

`0.85x` 只是允许下限，不是异常发生后的默认倍率，也不是固定档位。算法计算结果为 `0.934x` 时，目标就是 `0.934x`；只有计算结果接近 `0.85x` 时才可能下降到 `0.85x`。

## 二、“无感”的可验收定义

“无感”不能只解释为用户没有看到一个开关变化，必须同时满足：

1. 未介入时播放器行为与关闭该功能完全一致。
2. 不关闭或重建 tunneling、音频直通、硬解、SurfaceView、电视刷新率匹配或分辨率直出。
3. 不因为固定斜坡过慢而错过救援窗口并进入重缓冲。
4. 不因瞬时异常、单次重缓冲或旧趋势直接拉低倍率。
5. 倍率恢复时不频繁往返，不制造音频节奏抖动。
6. 字幕、音画同步、播放记录、Seek、选集和用户手动倍速保持原有语义。
7. 如果物理条件无法在 `0.85x` 以上稳定，必须明确判定为“保护能力不足”，不能宣称已经无感解决。

需要诚实说明：公开研究并不能证明 `0.85x` 对所有人、所有内容完全不可感知。熟悉人声、强节拍音乐、体育运动、横移镜头和原生帧率电视输出更容易察觉。本文所说的“无感”首先是播放体验层面不出现由控制器设计造成的卡顿、功能变化和突兀跳速；`0.85x` 的主观不可感知性仍必须通过真机内容分类测试验证。

## 三、不可突破的项目边界

1. Go 二进制代理必须保留。
2. `127.0.0.1` HTTP 播放链路必须保留。
3. App 不增加网盘 SDK/CDN 直连和鉴权能力。
4. 本地代理读取速度不得冒充真实网盘上游吞吐。
5. 用户启用 tunneling 或音频直通时控制器休眠，不得擅自关闭。
6. 电视输出模式继续按现有逻辑工作，网络保护不能接管 Display.Mode。
7. 用户手动倍速不是网络保护输入，用户设置始终优先。
8. 仅 EXO 点播使用；直播、MPV 和外部播放器不自动介入。

## 四、公开资料与开源实现结论

### 4.1 Media3：比例控制、平滑、安全边界和一秒更新

来源：

- https://github.com/androidx/media/blob/main/libraries/exoplayer/src/main/java/androidx/media3/exoplayer/DefaultLivePlaybackSpeedControl.java

可借鉴点：

- 使用比例控制器计算连续速度，不使用固定几个目标档位。
- 默认最短速度更新时间为 1 秒，而不是十几秒才调整一次。
- 使用死区避免微小噪声导致频繁 rate change。
- 对最小可达状态做指数平滑，并使用偏差安全边界。
- 重缓冲后调整控制目标，而不是把一次重缓冲直接映射到最低倍率。
- 默认 `0.97～1.03x` 是直播延迟控制的保守工程范围，但 Media3 允许业务设置其他上下限。

### 4.2 dash.js LoL+：缓冲危险程度决定调节强度

来源：

- https://github.com/Dash-Industry-Forum/dash.js/blob/development/src/streaming/controllers/CatchupController.js
- https://github.com/Dash-Industry-Forum/dash.js/blob/development/src/core/Settings.js

可借鉴点：

- LoL+ 在缓冲低于目标水位时使用非线性函数计算播放速度。
- 缓冲越危险，速度变化越积极；不是固定步长。
- 设置开始/停止窗口形成滞后，防止控制器在阈值附近振荡。
- stalled、paused、seeking 等状态具有独立处理，不把异常状态的旧样本继续用于计算。
- 小于最小有效变化时不反复写 playbackRate。

### 4.3 hls.js：只有安全条件成立才调速

来源：

- https://github.com/video-dev/hls.js/blob/master/src/controller/latency-controller.ts

可借鉴点：

- 先确认 forward buffer 仍有安全余量，再改变 playbackRate。
- 使用连续非线性函数，而不是异常即跳固定倍率。
- 离开有效控制区后恢复 `1.00x`。
- stall 会提高目标延迟，本质上是承认当前安全缓冲不足。

### 4.4 WebRTC NetEQ：正常调整与紧急调整分级

来源：

- https://github.com/webrtc-mirror/webrtc/blob/main/modules/audio_coding/neteq/decision_logic.cc

可借鉴点：

- 使用过滤后的当前缓冲与目标缓冲比较。
- 正常状态使用 accelerate/preemptive expand，严重偏离时使用 fast accelerate。
- 时间拉伸操作有最短间隔和状态约束。
- 解码恢复前要求缓冲达到最低水位，避免刚恢复就再次耗尽。

我们的方向与 NetEQ 相同：应按风险期限区分普通斜坡和紧急斜坡，但不能无条件进入最低倍率。

### 4.5 自适应媒体播放论文

- Smooth Control of Adaptive Media Playout for Video Streaming
  DOI：`10.1109/TMM.2009.2030543`
- Online Buffer Fullness Estimation Aided Adaptive Media Playout for Video Streaming
  DOI：`10.1109/TMM.2011.2160158`
- Subjective Assessment of Adaptive Media Playout for Video Streaming
  DOI：`10.1109/QOMEX.2019.8743320`
- Human Perception of Altered Video Speed
  DOI：`10.1163/22134468-bja10116`

共同结论：

- 自适应播放速度必须围绕缓冲 fullness、underflow 风险和连续控制设计。
- 平滑不等于越慢越好；过慢而错过 underflow 截止时间同样是错误控制。
- 主观感知与内容、动作、人声、音乐和个体有关。
- 既往工作可能低估较大速度偏差对体验的影响，不能把 `0.85x` 宣称为普遍完全无感。

## 五、物理模型

### 5.1 可支撑倍率

缓冲媒体时间的变化率可直接反映“生产速度与消费速度”的差：

```text
bufferSupported = currentSpeed + d(bufferSeconds) / d(wallClockSeconds)
```

例：

- 当前 `1.00x`。
- 20 秒内缓冲下降 1.4 秒。
- 缓冲斜率为 `-0.07 秒/秒`。
- 当前可支撑倍率约为 `0.93x`。

对 Go/loopback，这是主判断信号，因为它不依赖 App 具备上游网盘鉴权，也不会误把本地 socket 吞吐当成 CDN 吞吐。

### 5.2 可信网络估计融合

仅当路由为 `DIRECT_REMOTE_HTTP` 且媒体码率置信度足够时：

```text
networkSupported = trustedBandwidth * safetyFactor / mediaBitrate
supported = min(bufferSupported, networkSupported)
```

路由为 Go/loopback 时：

```text
supported = bufferSupported
```

媒体元数据码率通常是平均值，VBR REMUX 的瞬时消费可能明显更高。面板动态观测码率也不能直接成为控制器唯一输入；缓冲趋势仍是最终闭环证据。

### 5.3 安全缓冲线

不能只计算“距离缓冲为零还有多久”，必须保留恢复和请求抖动空间：

```text
safeBuffer = max(
    当前自动重缓冲恢复阈值,
    基础安全水位,
    已知请求抖动预算
)
```

第一版建议：

- 基础安全水位不低于 15 秒。
- 高码率/网盘/Go 路由优先使用 20～30 秒。
- 若未来 Go 提供请求 P95/P99，只读 telemetry 可把安全水位扩展为 `rebufferThreshold + 2 × requestP95`。

### 5.4 到达安全线的剩余时间

当缓冲斜率为负时：

```text
timeToReserve = (buffered - safeBuffer) / -bufferSlope
```

若 `buffered <= safeBuffer`，控制器已经进入紧急区，不能再使用普通慢速确认逻辑。

## 六、最终控制算法

### 6.1 资格门控

只有全部满足才观察或介入：

| 条件 | 要求 | 不满足时 |
|---|---|---|
| 功能开关 | 自动判断已允许 | 休眠 |
| 播放核心 | EXO | 休眠 |
| 内容 | 点播 | 休眠 |
| 用户速度 | `1.00x` | 用户速度优先 |
| 调速命令 | `COMMAND_SET_SPEED_AND_PITCH` 可用 | 休眠 |
| tunneling | 用户未启用 | 保留 tunneling，休眠 |
| 音频直通 | 用户未启用 | 保留直通，休眠 |
| 播放状态 | READY、playing、非 Seek | 仅保留状态，不采旧趋势 |
| Go 代理 | 保留原链路 | 不绕过、不直连 |

### 6.2 趋势估计

- 快 EWMA 用于快速发现缓冲下降。
- 慢 EWMA 用于抑制短时抖动和虚假恢复。
- 风险判断取两者较保守的结果。
- loader idle、满缓冲、暂停、Seek、切集、轨道切换时清除或冻结无效样本。
- 重缓冲可提高风险置信度，但没有新趋势时不得直接改变倍率。

### 6.3 连续目标倍率

```text
rawTarget = supported - safetyMargin
targetSpeed = clamp(rawTarget, 0.85, 1.00)
```

建议安全裕量初值为 `0.002～0.005x`，最终以真机日志校准。

目标示例：

| supported | 目标示例 | 解释 |
|---:|---:|---|
| 0.995 | 0.992～0.993 | 轻量保护 |
| 0.972 | 0.968～0.970 | 接近轻量区边界 |
| 0.936 | 0.931～0.934 | 深度保护，但不是最低倍率 |
| 0.853 | 0.850 | 计算结果才触及下限 |
| 0.810 | 不可稳定 | `0.85x` 也无法长期消除缺口 |

### 6.4 深度保护进入条件

进入 `0.85～0.97x` 必须同时满足：

1. `supported < 0.97x` 且 `supported >= 0.85x + safetyMargin`。
2. 快慢趋势均支持持续缺口，不能只有一个瞬时负样本。
3. 正常场景至少经过稳定确认窗口。
4. `timeToReserve` 已进入保护视野，或者已有重缓冲证据但恢复后出现了新的负趋势。
5. 期限可行性检查通过。
6. 当前没有 tunneling、直通、用户倍速或不支持保调等阻断条件。

### 6.5 期限可行性检查

下降前先计算在允许的平滑变化速度下，能否在安全缓冲线前到达目标：

```text
requiredRateChange = currentSpeed - targetSpeed
availableRampTime = max(0, timeToReserve - reserveMargin)
requiredSlew = requiredRateChange / availableRampTime
```

判定：

- `requiredSlew` 在允许范围内：可以介入。
- 只有使用深度保护斜坡才能及时到达：进入深度保护。
- 即使最快允许斜坡也来不及：标记“救援窗口不足”，不能继续用慢斜坡假装保护。
- `targetSpeed < 0.85x`：标记“带宽低于保护下限”。

该检查保证控制器不会因为固定 `0.008x/10秒` 而在达到有效倍率前耗尽缓冲。

### 6.6 期限驱动的动态斜坡

健康观察期可保持低频采样；介入期建议按 Media3 的工程做法缩短到约 1 秒更新一次。

每次调整量同时考虑比例误差和截止时间：

```text
error = currentSpeed - targetSpeed
proportionalStep = Kp * error
deadlineStep = error * updateInterval / max(availableRampTime, updateInterval)
step = max(proportionalStep, deadlineStep, minimumEffectiveChange)
nextSpeed = max(targetSpeed, currentSpeed - step)
```

同时设置每秒最大 slew 和变化加速度限制：

- 缓冲充足、目标仍在 `0.97～1.00x`：使用较慢斜坡。
- `timeToReserve` 缩短：自动提高斜坡速度。
- 进入 `0.85～0.97x`：仍趋近算法目标，不跳到下限。
- 如果截止时间要求突破最大允许 slew，则判定不可无感救援，而不是继续慢调直到重缓冲。

具体 `Kp`、最大 slew 和 reserveMargin 必须通过仿真与真机日志确定，不能凭固定档位拍脑袋。

### 6.7 进入滞后与恢复滞后

进入：

- 正常风险：趋势和风险持续确认，避免瞬时抖动误触发。
- 紧急风险：允许缩短确认，但至少需要连续多个有效样本，不能依据单点。
- 单次重缓冲只能缩短信任建立时间，不能直接指定目标倍率。

恢复：

- `supported` 恢复到接近 `1.00x`。
- 缓冲重新高于安全水位。
- 条件持续 20～30 秒后才逐渐回升。
- 恢复速度慢于进入速度，避免刚恢复就重新耗尽。
- 到达 `1.00x` 后退出保护，继续观察。

### 6.8 生命周期

| 事件 | 行为 |
|---|---|
| 用户暂停 | 恢复用户速度，清除趋势 |
| Seek/跳集/切轨 | 恢复 `1.00x`，清除旧样本，重新观察 |
| 首帧前 | 不依据不完整趋势降速 |
| 重缓冲开始 | 保留当前保护倍率，清除旧趋势 |
| 重缓冲结束 | 等新趋势，重缓冲只作为风险先验 |
| 停止/结束/新媒体 | 完整重置 |
| 用户手动倍速 | 自动控制退出 |
| 中途启用 tunneling/直通 | 平滑恢复 `1.00x` 后休眠 |

### 7.2 必须修正

| 当前 item-62 行为 | 风险 | 下一版修正 |
|---|---|---|
| 下限为 `0.97x` | 无法覆盖用户要求的较大但仍可通过降速补偿的缺口 | 下限扩展到 `0.85x`，保留 `0.97x` 优先区 |

### 7.3 需要补充的输入与诊断

下一版 Decision/日志至少增加：

- `safeBufferMs`
- `timeToReserveMs`
- `rawTargetSpeed`
- `calculatedTargetSpeed`
- `requiredSlewPerSecond`
- `appliedSlewPerSecond`
- `rampFeasible`
- `protectionTier`：light/deep
- `infeasibleReason`：below-floor/deadline-too-short/insufficient-confidence

没有这些数据，真机出现重缓冲时无法判断是估算错误、进入过晚、斜坡过慢，还是物理带宽确实不可解。

### 7.4 与自动缓冲策略联动

当前项目已有 `AutoLoadControl`、动态重缓冲恢复阈值和时间优先缓冲。网络保护不能独立使用固定安全线，应读取当前有效重缓冲恢复阈值形成 `safeBuffer`。

需要注意：

- 网络保护只改变消费速度，不提升 Go/CDN 吞吐。
- 自动缓冲负责建立安全库存，速度控制负责轻微缩小长期消费缺口。
- 两者必须共享重缓冲和缓冲水位信息，但不能互相反复改参数形成振荡。

### 7.5 电视直出的新增风险

代码可以做到“不关闭电视直出”，但播放速度低于 `1.00x` 后，媒体帧呈现节奏本身已经改变。例如 23.976fps 内容以 `0.85x` 播放时，有效内容节奏约为 20.38fps，而电视仍可能工作在 23.976Hz。

这可能出现：

- 重复帧或不均匀 cadence。
- 横移和体育镜头更容易察觉拖慢。
- 即使 Display.Mode 没被修改，也不能宣称电视直出体验完全不变。

因此真机验收必须单独覆盖：

- 手机屏幕。
- 电视原生 23.976/24Hz 直出。
- 50/60fps 体育或高运动内容。
- 字幕、人声和强节拍音乐。

如果电视深度保护区的 cadence 明显不可接受，应增加资格策略：电视帧率直出会话只允许轻量区，或者由用户明确允许深度区；不能偷偷关闭电视直出解决。

### 7.6 音频和设备负载

`0.85x` 需要更强的 Sonic 时间拉伸，可能增加：

- PCM 音频处理开销。
- 某些设备 AudioTrack underrun 风险。
- 人声间隔和音乐节拍感知。

后续可把 AudioTrack underrun、掉帧和热状态加入退出条件。第一版至少应记录这些指标，不能只看重缓冲次数。

## 九、必须新增的自动化测试

1. 单次重缓冲、无新趋势：保持 `1.00x`。
2. supported=`0.934x`：目标接近 `0.93x`，不得直接变成 `0.85x`。
3. supported=`0.853x`：只有持续证据和期限可行时才逐渐到达 `0.85x`。
4. supported=`0.81x`：判定低于保护下限，不宣称可稳定。
5. 缓冲充足：使用慢斜坡。
6. 缓冲接近安全线：动态加快斜坡，并在仿真中早于安全线到达目标。
7. 最大允许 slew 仍来不及：返回 deadline-too-short，不继续固定慢调。
8. 网络恢复：滞后确认后回到 `1.00x`。
9. Go/loopback：不使用本地带宽估计。
10. DIRECT_REMOTE_HTTP：网络估计只能让判断更保守。
11. tunneling/直通/用户倍速：始终休眠且不改变原功能。
12. Seek/切集/暂停：恢复用户速度并清除趋势。

## 十、真机验收标准

### 功能保持

1. tunneling 开启时保持原逻辑，网络保护休眠。
2. 音频直通开启时保持原逻辑，网络保护休眠。
3. 电视刷新率/分辨率切换仍正常。
4. Go 二进制和 `127.0.0.1` 链路不变。

### 健康网络

1. 全程保持 `1.00x`。
2. 不出现无意义 rate change。
3. CPU、掉帧和 AudioTrack 状态不退化。

### 可稳定的轻微缺口

1. 自动计算 `0.97～1.00x` 内目标。
2. 缓冲在到达安全线前停止下降。
3. 无新增重缓冲。
4. 恢复后自动回到 `1.00x`。

### 可稳定的深度缺口

1. 目标由算法计算，例如 `0.934x`，不直接跳到 `0.85x`。
2. 动态斜坡在 `timeToReserve` 截止前到达有效范围。
3. 不因为斜坡过慢进入重缓冲。
4. 检查人声、音乐、字幕、运动画面和电视 cadence。

### 不可稳定缺口

1. supported 低于 `0.85x` 时明确记录不可稳定。
2. 不无限降低倍率。
3. 不伪造“已保护成功”的状态。

## 十一、收益与限制

### 预期收益

- 覆盖比 item-62 更大的持续吞吐缺口。
- 目标倍率按真实支撑能力计算，不依赖固定档位。
- 避免固定慢斜坡在达到有效速度前耗尽缓冲。
- 对 VBR、高码率和 Go 代理场景仍使用缓冲闭环，不依赖错误的本地吞吐。
- 网络恢复后自动退出，不整场固定慢放。

### 明确限制

- `0.85x` 不能保证对所有内容和用户完全不可感知。
- 实际上游低于媒体消费量 15% 以上时，单靠降速无法长期稳定。
- 突然断网、CDN 长时间无数据、Go 请求阻塞和初始缓冲已经接近零时，不存在同时保证不变速感知和不重缓冲的算法。
- 降速不能替代更稳定的上游吞吐、更大安全缓冲或低码率版本。
- 电视原生帧率直出下，深度降速可能改变 cadence，必须真机验证。

## 十二、后续可选增强

1. Go 二进制提供只读 telemetry：上游读取字节、Range 请求时延、重试、P50/P95/P99 和上游 EWMA。
2. 动态安全缓冲结合 Go 请求 P95/P99，而不是固定秒数。
3. 记录 Sonic/AudioTrack underrun、温度和掉帧，作为深度保护退出条件。
4. 建立设备、ROM、音频输出和电视模式兼容矩阵。
5. 使用离线回放日志仿真控制器，不在每次用户播放时自动做 A/B 测试。

以上增强不得绕过 Go 代理，也不得改变用户已启用的输出功能。
